docmcp

Your docs, inside your coding agent.

A self-hosted MCP server that gives coding agents — OpenAI Codex and other MCP clients — search-and-cite access to your internal documentation. It runs entirely on your own infrastructure: no documents and no queries ever leave your network.

Apache-2.0 Model Context Protocol Self-hosted Runs fully offline
What you can do

One server, six superpowers for your docs

Point a coding agent at docmcp and it can find, read, and cite your internal knowledge — under access control you define, on infrastructure you control.

Ask in plain language

“Find the firmware flashing runbook and summarize it.” The agent searches, reads the right file, and cites the exact path and line.

Ingest almost anything

PDFs, Word/PowerPoint/Excel, HTML, Markdown, and source code become a clean Markdown store — tables and scanned pages included, via Docling + tree-sitter.

Control who sees what

Each user gets a bearer token scoped to path prefixes. Reusable groups keep grants tidy — a reader can never read outside their scope.

Search two ways

Fast keyword / full-text search by default (ripgrep · FTS5), plus an optional semantic vector layer (Qdrant + ONNX embedder) that stays fully offline.

Run it entirely on-prem

Models are vendored in the repo; ingest and serving need no external API and no internet — ideal for air-gapped or privacy-sensitive environments.

Operate from the browser

A setup wizard + admin console gets you running with no token to paste; an optional upload portal lets non-technical teammates publish docs without git.

How it works

Architecture at a glance

An offline pipeline builds a curated store; a hardened HTTP edge serves it to agents. Only the proxy is ever exposed to the network.

Setup

Up and running in three steps

The only thing the host needs is Docker (with the Compose plugin). The helper and the web console run everything in containers — no Python, no uv, no ripgrep on the host.

1

Get the code

Models ship as Git LFS data — install git-lfs first, then verify the pull.

# once per machine, before cloning
git lfs install
git clone <repo-url> && cd <repo>
./docmcp.sh models   # verify
2

Launch the server

The browser console opens a setup wizard — no token to paste on first run.

./docmcp.sh console \
   --docs /path/to/your/docs

# …or the scriptable CLI:
./docmcp.sh setup
./docmcp.sh ingest && ./docmcp.sh serve
3

Connect your agent

Mint a scoped token and point Codex at the server. Keep the token in an env var.

# operator mints a scoped token
./docmcp.sh token alice /public /team-fw

# user registers the MCP server
export DOCS_MCP_TOKEN=tok_alice_xxxx
codex mcp add docs \
  --url http://10.0.0.5/mcp \
  --bearer-token-env-var DOCS_MCP_TOKEN
Network profile. The shipped default is the internal-network profile — plain HTTP over a trusted network such as a VPN (bearer tokens travel unencrypted, so keep it on a network you trust). For a public network set DOMAIN=docs.company.internal and Caddy serves automatic HTTPS. Full profiles, ports, and the security model live in docs/OPERATING.md.
After setup

Day-two operations

The two things you'll actually do once it's running: grow the corpus, and survive a reboot.

Add more documents, anytime

No need to take the server down. The recommended way to add a folder is to link it — ingest reads it in place, no copy — then rebuild the index (it's incremental, and new docs are searchable as soon as ingest finishes).

./docmcp.sh link /path/to/more/docs  # register — read in place, no copy
./docmcp.sh ingest                    # rebuild the index

./docmcp.sh add <path> instead copies into raw/ (handy for a single file, but duplicates storage — prefer link for folders). Also: the console's Ingest page (one click), ./docmcp.sh schedule 30m for auto-ingest, or the optional upload portal for non-technical teammates.

It survives reboots

The serving stack runs with restart: unless-stopped and your data lives in Docker volumes plus repo files — a reboot loses nothing, and the server comes back on its own (no serve re-run needed).

On a Mac laptop the one catch is Docker Desktop — it doesn't launch at login unless you enable “Start Docker Desktop when you sign in.” Once Docker is up the stack self-heals; confirm with ./docmcp.sh status. On a Linux server it's fully automatic. (Only ./docmcp.sh uninstall ever deletes your data.)

The interface

Four tools the agent calls

Every tool intersects paths with the caller's allowed prefixes; read_doc denies (not silently empties) a disallowed path.

ToolReturnsNotes
list_docsindex entries under a pathpre-filtered to your prefixes
search_docs{path, line, snippet, score}keyword — ripgrep / FTS5
read_doc{path, content, …}denies paths outside your prefixes
semantic_search{path, line, snippet, score}off unless ENABLE_VECTOR=true
