# Upload/manage portal (optional)

A browser portal where teammates **publish and manage documents without git**. It is a
**separate service** that writes **only** to `raw/` (the ingest source) — it never runs
Docker and never touches the curated store, so `docs-mcp` stays strictly read-only. The
existing cron `schedule` ingests what the portal writes, so uploads become searchable on
the next tick.

> **Optional.** If your contributors are technical git users, a git-PR flow is simpler and
> adds zero new attack surface. Build the portal only when non-technical folks must upload.

## Security model (read this first)

- **Write needs an explicit grant.** Only tokens minted with `--write <prefix>` can publish,
  and only into those prefixes. Write scope is **separate** from read (`allowed_prefixes`),
  so a reader is never a writer and the admin read-all token is not a write-anywhere token.
- **Session cookies need a trusted network.** Login exchanges your `tok_…` for an
  HMAC-signed, `HttpOnly` + `SameSite=Strict` cookie (the bearer token itself never enters
  the cookie). Over plain HTTP the cookie is **not encrypted**, so set
  `ALLOW_PLAINTEXT_PORTAL=true` only on a trusted VPN; for a public network use `DOMAIN=<host>`
  so Caddy serves HTTPS.
- **Containment + RBAC on every write.** Every upload/rename/move/delete is canonicalized
  under the staging root (rejecting `..`/NUL/abs/symlink escapes) and then RBAC-checked on the
  *resolved* path against your `writable_prefixes`. Uploads are validated (type allowlist, size
  cap, Git-LFS-pointer rejection) before they are written. State-changing actions require a CSRF
  token. Every action is recorded in an append-only audit log (never the bearer token).

## Enable it

```bash
# 1) In .env (./docmcp.sh setup already generated SESSION_SECRET):
#    PORTAL_ENABLED=true
#    ALLOW_PLAINTEXT_PORTAL=true        # plaintext on the VPN (or set DOMAIN=<host> for HTTPS)
#    SESSION_SECRET=<auto-generated>    # HMAC key for session cookies

# 2) Grant a teammate WRITE access to a folder (read + write here):
./docmcp.sh token alice /team-fw --write /team-fw --expires 90d
#   write-only "dropbox" (can publish, cannot read):  token bot --write /incoming
#   read everywhere, write one folder:                token lead --all --write /team-fw

# 3) Make uploads auto-publish (the portal does NOT ingest; the cron does):
./docmcp.sh schedule 5m

# 4) Start everything (serve brings the portal up when PORTAL_ENABLED=true):
./docmcp.sh serve
```

Reach it at **`http://<server-ip>/portal`** (or `https://<domain>/portal`). Sign in with the
same `tok_…` used in Codex.

## What users can do

- **Upload** files (PDF/DOCX/PPTX/Markdown/text/code…) into a folder they may write.
- **Manage**: rename, move (between writable folders), and delete their files — every endpoint
  is RBAC-checked on both source and destination. Rename keeps the file type inside the ingest
  allowlist, and rename/move **refuse to overwrite** an existing file unless you tick
  *overwrite* (which first snapshots the file being replaced into version history).
- **Versioning**: re-uploading — or overwriting via rename/move — an existing path keeps the
  previous copy under `raw/.portal/versions/` (a dotdir the ingest skips); the latest is live.

## Operate

```bash
./docmcp.sh token-list                 # see who has write= scope
cat raw/.portal/audit.jsonl            # portal upload/manage audit (never the token)
./docmcp.sh status                     # last ingest time (when uploads went live)
```

Auto-publish latency = the cron interval. To lower it, run `schedule` more often; the
incremental ingest skips unchanged files, so frequent ticks are cheap.

## Limits / knobs (`.env`)

- `MAX_UPLOAD_BYTES` (default 25 MiB/file), `MAX_UPLOAD_FILES` (20/request).
- The portal mounts `raw/` **read-write** and `tokens.json`/`groups.json` read-only — and
  nothing else (no Docker socket, no curated store).
